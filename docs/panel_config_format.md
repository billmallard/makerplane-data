# Panel config format — what the editor saves, what the device runs

This explains the file the configuration manager produces when you lay out a
panel, and how it maps to pyEfis. The short version: **the editor saves native
pyEfis config.** There is no separate "design" format and no compile step — the
YAML the editor stores is (almost exactly) the YAML the device loads.

Related: [device_deployment.md](device_deployment.md) (how it reaches the Pi),
[../configurator/README.md](../configurator/README.md) (the app),
[system_designer.md](system_designer.md) (the vision).

## Why no compile step

Earlier drafts had the editor store its own "design" JSON and a server-side step
"compile" it into pyEfis YAML. That's an extra moving part with nothing to gain:
the editor already works in pyEfis's own grid + instrument model, so it just
emits pyEfis config directly. Benefits:

- **One source of truth.** The bytes you see in the editor's Code pane are the
  bytes the device runs. No translation to drift out of sync.
- **The Code pane is real pyEfis config** — editable by hand, copy-pasteable into
  a `screens/` file, reviewable in a PR.
- **Delivery is a file copy**, not a build (see [device_deployment.md](device_deployment.md)).

The only transforms the editor does are mechanical and happen at save time in the
browser (described below): expand element-groups, and strip a couple of
editor-only keys.

## The artifact

A panel is a **pyEfis config fragment**: a `main` block (screen size + which
screen boots) and one screenbuilder `screen`. Example (a small panel):

```yaml
main:
  screenWidth: 1280
  screenHeight: 800
  defaultScreen: PANEL
  screenDiagonalInches: 10.1      # editor metadata; pyEfis ignores unknown main keys
screens:
  PANEL:
    module: pyefis.screens.screenbuilder
    title: Panel
    layout:
      rows: 110
      columns: 200
    instruments:
      - type: airspeed_tape
        row: 6
        column: 4
        span: { rows: 98, columns: 24 }
      - type: atitude_indicator
        row: 6
        column: 68
        span: { rows: 64, columns: 64 }
        options:
          symbol_color: "#ffff00"
      - type: altimeter_tape
        row: 6
        column: 172
        span: { rows: 98, columns: 24 }
```

This is exactly a pyEfis screenbuilder screen plus a `main`. Drop the `screens:`
entry into a device's `screens/` config and set `main.defaultScreen` and it runs.

## Field mapping (editor → pyEfis)

| Editor concept | pyEfis field |
|---|---|
| target resolution (e.g. 1280×800) | `main.screenWidth` / `main.screenHeight` |
| which screen shows on boot | `main.defaultScreen` (always `PANEL`) |
| physical screen size (DPI hint) | `main.screenDiagonalInches` — **editor-only**, pyEfis ignores it |
| the canvas | `screens.PANEL` (`module: pyefis.screens.screenbuilder`) |
| the grid | `screens.PANEL.layout.rows/columns` (always **110 × 200**) |
| each placed widget | an entry in `screens.PANEL.instruments` |

### The grid

pyEfis screenbuilder uses a **normalized 110-row × 200-column grid** — it is the
same regardless of the physical resolution (the layout engine scales the grid to
the real pixels). So a panel laid out at one resolution maps cleanly to another;
the *aspect ratio* is the meaningful variable, which is why the editor targets a
resolution and reflows the canvas to its aspect.

### Instruments

Each instrument is `{ type, row, column, span: { rows, columns }, options }` —
identical to a hand-written screenbuilder instrument. `type` is the pyEfis
widget name (`airspeed_tape`, `atitude_indicator`, `arc_gauge`, …; the full set +
each type's options is the schema the editor loads from
`/assets/editor/schema.json`, generated by `pyefis.editor.schema`). `options` is
omitted when empty. Grid numbers are rounded to one decimal.

## Element groups are expanded

The editor offers **element groups** (e.g. a 4-cylinder CHT/EGT cluster) you drag
in as one unit and can scale together. pyEfis has **no group concept** — a screen
is a flat list of instruments. So at save time the editor **expands** each group
into its child instruments at absolute grid coordinates (the same math as the
editor's *Ungroup*): a child at local grid position within the group's box is
placed at `group.row + child.row/group.gridRows * group.span.rows` (and likewise
for column/spans). The saved YAML therefore contains only concrete instruments.

Consequence: groups are an editing convenience that lives in the canvas, not in
the file. A group still scales/ungroups on the canvas, but the moment the config
is written (or you edit the Code pane) it is flat instruments — re-opening shows
them individually. That's intentional: the file is pure pyEfis.

## Editor-only options that are stripped

`preview_scene` (on `virtual_vfr`) only selects which synthetic-vision *preview
image* the editor shows; it has no meaning on the device and is removed on save.

## Round-trip + backward compatibility

The editor reads the same format back when you open a device. For older devices
saved before this change, it also accepts the **legacy "design" form**
(`{ kind: pyefis-screen, screen, layout, instruments }`) so they still open; the
next save rewrites them in the native form. (`fromStoredDoc()` /
`toPyefisDoc()` in `public/editor.html`.)

## How it reaches the device

The saved YAML is stored per device (R2 + a versioned `configs` row) and pulled
by the paired device:

```
GET /device/config            Authorization: Bearer <device token>
  -> 200  application/x-yaml   + ETag: "v<n>"  + X-Config-Version: <n>
  -> 304  (when If-None-Match matches the current version — a cheap poll)
  -> 404  no config yet
  -> 401  missing / unknown token
```

Each pull stamps `last_pull_at`. The on-Pi updater then atomic-swaps the screen +
`main` into the device's config (see [device_deployment.md](device_deployment.md),
P3). Because the artifact is already pyEfis config, "install" is a file drop, not
a build.

## Known limitations / follow-ups

- **Synthetic vision data paths.** `virtual_vfr` renders SVS terrain only when its
  `options.svs` block points at the on-device terrain/airport/water data. The
  editor doesn't know a given Pi's data layout, so it does **not** inject those
  paths yet; the widget will fall back to sky/ground attitude until the device's
  SVS config is wired (planned as a device-side `svs` include rather than
  hard-coding paths per panel). The panel *layout* is fully delivered regardless.
- **No `include:` emitted.** The editor places concrete instruments, so it never
  emits pyEfis `include,…` instruments. Stock screens that use includes still
  work on the device; the editor simply doesn't generate them.
- **Aircraft limits / V-speeds / gauge bands** are not in the panel config — they
  come from the FIX database / aircraft config (a separate concern, fix-gateway
  side). The panel only describes *which* instruments go *where*.
